## SOURCE MATERIAL

### From Tool to Product

The lar-platform pipeline works. Two platforms prove it. But right now, it requires an architect who understands the thesis — someone who can guide the AI through the 10 prompts, validate the domain decomposition, and refine the output.

Bridge Builder is the product that democratizes this process.

**The vision:** A SaaS platform where you describe your domain in plain English and get a production-grade, architecturally sound platform back. Not vibe code that unravels at month 3. Platform code that scales with your business.

The pipeline that generated PFM-Platform (11 domains, 148 components), TP-Platform (8 domains, Supabase), and TRN-Platform (4 domains, SQL Server) becomes the engine of a product. Your domain in. Your platform out.

### The POC Sequence

Bridge Builder is being built in stages. Each POC adds a capability:

**POC 1: Domain Decomposer** — COMPLETE
The first proof-of-concept: a tool that takes a domain description and outputs a decomposition map. Bounded contexts, entities, relationships, API surfaces. This is Prompts 1-4 of the pipeline, wrapped in a structured interface.
Proven on TP-Platform: plain English → 8 domains in ~6 hours.

**POC 2: AI Chat Trip Planner**
A conversational interface for domain refinement. Fixed UI, but the data layer responds to natural language. "Add an EV charging domain." "Rename 'parks' to 'destinations'." "Show me the entity schema for itineraries."
This proves that domain modification can be conversational, not just prompt-based.

**POC 2.5: Sinew Orchestration**
Integration of the Sinew engine (formerly HooksAI). File watchers, chained workflows, permission gates.
When you speak a change, Sinew orchestrates the cascade: schema update → data-access regeneration → feature hook update → UI component refresh → Storybook story update → test re-run. All automated. All validated.

**POC 3: Red Pencil**
UI composition plus bidirectional generative cycle, running live.
Red Pencil is the editorial tool: mark up a page, annotate what should change, and the system generates the new version. The "pencil" produces "marks" that become "changes" that become "stories" that become "pages." The generative cycle runs in both directions.

**POC 3.5: Domain Evolution**
The pipeline can not only CREATE platforms — it can EVOLVE them.
Schema migration, versioning, and domain growth. TP-Platform's roadmap projects growth from 8 to 11 domains (adding EV support, weather integration, social sharing). The pipeline manages the evolution without breaking existing domains.

**POC 4: Bridge Builder SaaS**
The full product. GitHub integration. Worktree pattern for customer forks. Multi-tenant architecture.
Customers connect their vibe-coded repo. Bridge Builder analyzes it, proposes a decomposition, generates the platform architecture, and manages ongoing evolution. Customer forks (Layer 4) let different clients customize without diverging from the core.

### The Market

The vibe coding market is $4.7 billion with a 38% CAGR. Thousands of developers and entrepreneurs ship AI-generated apps every month. Most hit the Spaghetti Point at month 3.

Current solutions don't address the root problem:
- **Modernization services** (Orafox, Baytech) rewrite monoliths from scratch — expensive ($500K-$2M), slow (6-18 months), risky (50%+ project overrun rate)
- **AI code assistants** (Copilot, Cursor, Cline) generate more code faster — but more code without architecture is more spaghetti
- **Platform engineering tools** (Backstage, Port) assume you already have a platform to manage

Bridge Builder occupies a unique position: it **decomposes** rather than rewrites. It takes what you've built and gives it architecture. The spaghetti unwinds into layers.

### Full Circle: The Spaghetti Point, Resolved

Return to Episode 1. Month 3. The vibe-coded app has hit the wall. Features are slow. Bugs cascade. The team is afraid to touch the codebase.

Now there's an answer:

1. Connect Bridge Builder to the repo
2. The pipeline analyzes the code and proposes domain boundaries
3. The developer validates and refines (via AI Chat, POC 2)
4. Bridge Builder generates the layered architecture — typed packages, API routes, query hooks, UI components, Storybook stories, tests
5. Features migrate one by one from the monolith to the platform
6. The spaghetti unwinds. The architecture scales.

Not a rewrite. A decomposition. The code that exists is reorganized, not replaced. The AI that created the complexity now resolves it — because now the AI has been taught architecture.

### The Thesis, Restated

Vibe coding creates. Platform architecture sustains. Bridge Builder bridges the gap.

Features are spoken into existence, but they land in a structure that can grow. Components compose into workflow stories. Stories compose into pages. Pages fork for customers.

The Layers of Resolution thesis is not a theory. It is a proven pipeline with two working proofs and a third in progress:
- 148 components across 11 financial domains (PFM)
- 8 domains generated for trip planning (TP)
- 4 domains with a 4-layer SQL Server architecture (TRN)

The platform that presents its own origin story IS the proof. And Bridge Builder will bring that proof to everyone who needs it.
