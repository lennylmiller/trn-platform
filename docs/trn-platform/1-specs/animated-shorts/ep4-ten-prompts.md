## SOURCE MATERIAL

### What lar-platform Is (and Isn't)

lar-platform is not a framework you install. It's not a code generator that outputs boilerplate. It's a **decomposition toolkit** — a collection of 10 sequential AI prompts, JSON validation schemas, parameterized code templates, and annotated examples from the 148-component PFM platform.

The name stands for **Layers of Resolution** — the thesis from Episode 2, made operational.

It lives inside the pfm-platform repository as a directory: `/pfm-platform/lar-platform/`. It contains:
- `prompts/` — 10 sequential markdown files (01 through 10, plus 01.5 for brand discovery)
- `schemas/` — JSON Schema files for validating catalogs and manifests
- `templates/` — ~80 parameterized code templates with `{{placeholder}}` syntax
- `examples/` — Annotated real patterns from the PFM platform

Think of it as a recipe that teaches AI how to decompose a domain into a layered architecture. The kitchen (your codebase, your IDE, your AI tool) varies. The recipe is constant.

### The 10 Prompts

Each prompt builds on the output of the previous one. Sequential, not parallel. Constrained, not freeform.

**Prompt 1: Domain Discovery**
Input: A plain-English description of the application
Output: A list of bounded contexts — the natural domain boundaries
Example input: "A trip planning application where users create trips with itineraries, add destinations and activities, manage budgets per trip, coordinate with travel members, and track fuel costs for road trips."
Example output: 8 domains — trips, itinerary, parks, budget, members, fuel, todos, vehicles

**Prompt 2: Entity Extraction**
Input: The domain list from Prompt 1
Output: Zod schemas for every entity in every domain, with validation rules, types inferred via z.infer<>
The AI extracts entities from the domain descriptions and generates typed schemas. Every field gets a type. Every boundary gets validation.

**Prompt 3: Use-Case Generation**
Input: Entities from Prompt 2
Output: Business workflows extracted from the entities — the things users actually DO with the data
Example: "Create a trip" → requires trips entity + members entity + itinerary entity

**Prompt 4: Workflow Discovery**
Input: Use cases from Prompt 3
Output: Storybook workflow stories — multi-step user journeys mapped to use cases
Each workflow becomes a WF-numbered story with defined steps, entry points, and expected outcomes.

**Prompt 5: Component Catalog Generation**
Input: Entities, use cases, and workflows
Output: A machine-readable JSON catalog of every component needed — name, domain, category, props, description
This catalog IS the connective tissue from Episode 2. It's what makes the system AI-traversable.

**Prompt 6: Pages Catalog Generation**
Input: Workflows and component catalog
Output: A mapping of workflow steps to pages and component layouts
Which components appear on which pages, in what arrangement, with what data.

**Prompt 7: Test Strategy**
Input: All previous outputs
Output: Testing architecture per layer — Vitest config, coverage thresholds, MSW mock patterns, integration test shapes
80% coverage thresholds across lines, functions, branches, statements.

**Prompt 8: Infrastructure Scaffold**
Input: Domain structure, entity schemas
Output: CI/CD configs, database configs (Supabase or SQL Server), build system (tsup, pnpm workspaces), deployment templates
The infrastructure adapts to the target: Supabase for cloud, SQL Server + Express for enterprise.

**Prompt 9: Storybook Composition**
Input: Workflows, pages catalog, component catalog
Output: Storybook config, workflow stories, page stories, MSW mock handlers, decorators
Every component gets a co-located story. Every workflow gets a documented journey.

**Prompt 10: SaaS Shell Scaffold**
Input: All outputs
Output: Next.js + MakerKit + Turborepo starter with auth, billing, i18n, blog
The production shell that will consume the domain packages.

### The Trip Planner Example

The pipeline was tested on a trip planning application. Here's what happened:

**Input (plain English):** "A trip planning app where users create trips, build itineraries, find parks and destinations, manage travel budgets, invite members, track fuel costs, manage pre-trip todos, and register vehicles."

**Pipeline execution time:** ~6 hours of AI work with human domain verification

**Output:**
- 8 domains identified (trips, itinerary, parks, budget, members, fuel, todos, vehicles)
- 18 packages scaffolded in strict 3-layer architecture
- 125 tests passing
- 35+ Storybook stories
- 5 workflow stories with 19 steps and 4 full-flow demonstrations
- Navy + Amber theme with semantic design tokens
- Domain evolution roadmap projecting growth from 8 to 11 domains (including EV support)

The result is tp-platform — a fully structured monorepo that looks like it was built by a senior engineering team over months, generated in hours.

### The Constraint Principle

The most important design decision in lar-platform: **the AI is constrained by the architecture, not freed from it.**

In vibe coding, the AI generates whatever works. In the pipeline, the AI generates what BELONGS — the right code, in the right package, in the right layer, with the right imports.

The prompts don't say "generate a React component." They say "generate a data-access hook in the @tp-platform/budget-data-access package that wraps a Supabase query for the budget_entry table, validates the response with the BudgetEntrySchema from @tp-platform/shared, and uses the query key factory pattern from the data-access template."

Constraints produce coherence. Coherence scales. This is the opposite of spaghetti.
