## SOURCE MATERIAL

### The Proof Strategy

The thesis from Episodes 2 and 3: software complexity resolves in layers, and the pattern proven at 148 components can be extracted into a pipeline. The pipeline from Episode 4: 10 sequential prompts that decompose a domain into a platform.

Now the question: does it actually work? Not once (PFM was hand-built, not pipeline-generated). Not in theory. In practice. On different domains. With different backends.

The proof strategy is deliberately tiered:
- **Easy:** Trip Planner (familiar domain, small scope, Supabase backend)
- **Moderate:** Personal Finance (PFM — 11 domains, proven manually, validates the pattern)
- **Complex:** Health Insurance / QC (regulated industry — future proof)

Two of these proofs are complete.

### Proof 1: TP-Platform (Trip Planner)

TP-Platform was the first real test of the pipeline. A trip planning application — everyone understands the domain.

**What went in:** A plain-English description of a trip planning app (users, trips, itineraries, parks, budgets, members, fuel, todos, vehicles).

**What came out:**
- 8 domains identified and scaffolded
- 18 packages in strict 3-layer architecture (data-access → feature → ui-mui)
- Each package with its own package.json, TypeScript config, Vitest config
- 125 tests passing with coverage thresholds
- 35+ Storybook stories across all domains
- 5 workflow stories (19 steps total, 4 full-flow demonstrations)
- Navy + Amber theme with semantic design tokens
- Supabase backend with Row Level Security

**Time:** ~6 hours of AI work with human domain verification

**Backend:** Supabase (cloud-hosted PostgreSQL with auth, storage, realtime)

**The verdict:** The architecture held. Components were composable. Tests passed. The pipeline works on an easy domain.

### Proof 2: TRN-Platform (Training Platform)

TRN-Platform was the harder test. Three things changed simultaneously:

**1. Different backend:** SQL Server + Express instead of Supabase.
- Raw parameterized SQL via the `mssql` library instead of Supabase client calls
- An Express 5 server handling routes, middleware, validation
- This forced extending from 3 layers to **4 layers** — adding a `server/` package per domain
- Dual databases: `qc_training` for training content, `qc_core` for production reference data

**2. Different domain:** Training and execution, not CRUD.
- `steps` — a library of reusable executable steps (SQL queries, shell commands, manual pauses)
- `flows` — ordered sequences of steps with pause points and presenter notes
- `compositions` — stories, tutorials, and modules combining narrative blocks with executable flow blocks
- `execution` — real-time step execution via Server-Sent Events (SSE), with pause/resume

**3. Higher complexity:** The execution domain manages SSE connections, step orchestration, flow state, and error handling — far more stateful than typical CRUD operations.

**Architecture comparison:**

| Dimension | TP-Platform | TRN-Platform |
|-----------|-------------|--------------|
| Domains | 8 | 4 |
| Backend | Supabase | SQL Server + Express |
| Layers | 3 (da/feature/ui) | 4 (server/da/feature/ui) |
| UI framework | MUI 7 | MUI 9 |
| Real-time | Supabase Realtime | Server-Sent Events |
| Auth | Supabase Auth | Express middleware |
| Database access | Supabase client | Raw parameterized SQL |

**Shared DNA:** Both use pnpm workspaces, React 19, TanStack Query 5, TypeScript strict mode, Vitest, Storybook, Zod schemas, Emotion styling, tsup builds, and changesets for versioning. The pipeline adapts the target; it doesn't change the architecture.

### The Self-Referential Moment

Here is where it gets recursive.

TRN-Platform has a domain called `compositions`. Compositions are structured learning experiences — containers that hold:
- **Narrative blocks** — markdown prose (introductions, explanations, context)
- **Flow blocks** — embedded executable demos (run a flow live, watch SSE stream results)
- **Note blocks** — technical content (code snippets, architecture diagrams, SQL examples)
- **Composition blocks** — references to OTHER compositions (drill-in / drill-out navigation)

These compositions are used to deliver training presentations. A presenter opens a composition, navigates through blocks, runs live demos, and walks the audience through a learning journey.

Now: the Bridge Builder presentation — the one that tells the story of the pipeline (Acts 1-6) — is itself a set of compositions in the TRN-Platform database.

The presentation about the pipeline... runs on the platform the pipeline generated.

The narrative blocks explaining Layers of Resolution are stored in `composition_block` rows. The flow blocks that demo SQL execution are stored as `flow_id` references. The composition blocks that link Act 1 to Act 2 to Act 3 are stored as `ref_composition_id` entries.

**And there is a demo flow called "Composition Self-Query" that does this:**

Step 1: `SELECT * FROM composition WHERE title LIKE '%Bridge Builder%'`
→ Returns the row for the Bridge Builder module

Step 2: `SELECT cb.seq, cb.block_type, cb.heading FROM composition_block cb WHERE cb.composition_id = (SELECT TOP 1 composition_id FROM composition WHERE title LIKE '%Bridge Builder%') ORDER BY cb.seq`
→ Returns the block structure of the presentation being delivered

Step 3: (Manual pause) — Presenter says: "You are looking at the data structure that is currently driving this presentation."

The abstraction collapses. The presentation is not ABOUT data. The presentation IS data. And you can query it.

### Why This Matters

Self-reference is not a party trick. It's proof of architectural coherence.

If the platform can present its own architecture, query its own data, and execute its own demos, then every layer is working correctly:
- The server layer serves the composition data via Express routes
- The data-access layer fetches it via TanStack Query hooks
- The feature layer orchestrates navigation (drill-in/out, block sequencing)
- The UI layer renders it with MUI components and Storybook stories

Every layer had to work. Every contract had to hold. The self-referential demo only works if the architecture is sound end-to-end.

One platform is a project. Two platforms are a pattern. A platform that presents its own origin story is a proof.
